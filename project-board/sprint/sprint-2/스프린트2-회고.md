## KPT 회고

### Keep (계속할 것)

- **기술 부채 우선 해소** — 폴더구조·CI/CD·문서 보강을 먼저 해서, QA 점수·품질 게이트를 올리는 전략이 효과적이었음.
- **문서 한 곳에 매핑** — CLAUDE.md에 Phase별 실제 산출물 표를 넣어 두어, "규칙 vs 실제 파일"이 한눈에 들어옴.
- **Node 내장 테스트** — `node:test`로 단위 테스트 1곳 추가 시 별도 의존성 없이 테스트 전략 문구를 채울 수 있었음.
- **접근성·CI 실행 문서화** — WCAG 체크리스트, CI 실행 가이드를 만들어 두어 재현·검증이 쉬워짐.

### Problem (문제점)

- **Phase 2~4 진행 로그 미작성** — phase-log 가이드·템플릿은 추가했으나, 실제 Phase 2·3·4 완료 시점의 sprint-log는 아직 한 곳에 모아 두지 않음.
- **CI 실제 통과 이력** — ci.yml·deploy.yml은 존재하나, PR/푸시로 한 번이라도 녹색 통과한 Run이 있어야 "검증 완료"로 남김.
- **Should Have 미구현** — 다크 모드, 360px 반응형, 토스트 등은 스프린트 2 범위 밖으로 남음.

### Try (시도할 것)

- **PR 1회 생성 후 CI 통과 확인** — 로컬 lint·test 통과 후 push → PR 생성 → Actions에서 녹색 확인 후 문서에 "CI 검증 완료" 명시.
- **Phase 2·3·4 요약 로그 1회 작성** — phase-log 가이드 템플릿을 채워 `sprint-log.md` 또는 별도 파일에 "Phase 2~4 완료 요약" 한 블록 추가.
- **스프린트 3에서 Should Have 1~2개** — 다크 모드 또는 360px 반응형 중 하나 우선 적용해 완성도·UX 점수 보강.
