# 기술 스택 선택 근거

**프로젝트**: claude-commin-kit (AI 네이티브 개발 워크플로우 툴킷)
**작성일**: 2026-03-13
**스프린트**: Sprint 1

> 이 프로젝트는 **제로 의존성 서버리스 툴킷**입니다. 전통적인 프론트/백엔드/DB/인프라 구분 대신, 실제 구성 요소(브라우저 UI / CLI / 데이터 저장 / 배포)로 분류합니다.

---

## 1. 프론트엔드 (브라우저 UI)

### Pure HTML5 + CSS3 + Vanilla JavaScript

- **선택 이유**
  - 이 툴킷의 핵심 가치는 **"열면 바로 작동한다"**는 것입니다. 프레임워크를 쓰는 순간 `npm install`, 번들 빌드, Node.js 버전 맞추기 등 부대비용이 발생합니다.
  - 브라우저는 HTML/CSS/JS를 네이티브로 실행합니다. 빌드 없이 파일을 열면 즉시 작동하는 유일한 선택지입니다.
  - 외부 의존성이 없으면 공급망 공격(supply chain attack), 패키지 deprecated 위험, 버전 충돌이 원천 차단됩니다.

- **대안과의 비교**

  | 대안 | 장점 | 탈락 이유 |
  |------|------|----------|
  | React (Vite) | 컴포넌트 재사용, 상태 관리 편리 | 빌드 필수, `npm install` 필요, 번들 파일 관리 부담 |
  | Vue 3 | 직관적 문법, 경량 | 동일하게 빌드 필요, CDN 방식은 트리쉐이킹 불가 |
  | Svelte | 컴파일 결과가 Vanilla에 가까움 | 컴파일 단계 필수, 도구 체인 필요 |
  | Alpine.js (CDN) | 선언적 UI, 경량 | 외부 CDN 의존 → 오프라인 동작 불가 |

- **팀 역량**: 높음 — HTML/CSS/JS는 모든 웹 개발자의 기본 역량. 온보딩 비용 제로.

---

### CSS Custom Properties (CSS 변수)

- **선택 이유**
  - 빌드 도구 없이 테마(색상, 간격, 폰트) 시스템을 구현하는 유일한 네이티브 방법입니다.
  - JavaScript에서 `element.style.setProperty('--color', '#fff')`로 런타임에 테마 전환 가능합니다.
  - 모든 현대 브라우저에서 지원됩니다(Chrome 49+, Firefox 31+, Safari 9.1+).

- **대안과의 비교**

  | 대안 | 탈락 이유 |
  |------|----------|
  | Tailwind CSS | 빌드(PostCSS) 필수, CDN 버전은 용량 과대(~3MB) |
  | SCSS/SASS | 컴파일 필요, Node.js + sass 패키지 필요 |
  | CSS-in-JS | JS 런타임 오버헤드, 번들러 필수 |
  | 하드코딩 색상 | 테마 변경 시 파일 전체 수정 필요 |

- **팀 역량**: 높음 — CSS 표준 기능, 별도 학습 불필요.

## 2. 백엔드 (CLI — Node.js)

이 프로젝트에서 "백엔드"는 서버가 아니라 **터미널에서 실행하는 Node.js CLI**입니다.

### Node.js ESM (내장 모듈만 사용)

- **선택 이유**
  - `node:fs`, `node:path`, `node:child_process`, `node:readline` 등 Node.js 내장 모듈만으로 필요한 모든 기능(파일 읽기/쓰기, 경로 처리, Git 명령 실행, 터미널 입력)을 구현할 수 있습니다.
  - `package.json`이 없어도 `node kit/cli.mjs` 한 줄로 즉시 실행됩니다.
  - Node.js 18+ LTS는 ESM을 기본 지원합니다. Babel, ts-node 등 트랜스파일러 불필요.

- **대안과의 비교**

  | 대안 | 장점 | 탈락 이유 |
  |------|------|----------|
  | commander.js | CLI 옵션 파싱 편리 | 외부 패키지 의존성 발생 |
  | Python 스크립트 | 풍부한 표준 라이브러리 | Node.js 없는 환경에서도 Python 설치 필요, JS 개발자에게 생소 |
  | Bash 스크립트 | 추가 런타임 불필요 | Windows 호환성 문제, 복잡한 로직 구현 어려움 |
  | Deno | TypeScript 네이티브, 보안 모델 우수 | 별도 설치 필요, Node.js보다 생태계 작음 |

- **팀 역량**: 높음 — Node.js는 프론트엔드 개발자의 기본 환경. 내장 모듈이므로 문서 확인만으로 충분.

## 3. 데이터 저장

전통적 데이터베이스(PostgreSQL, MongoDB 등) 대신 **두 개의 저장 레이어**를 사용합니다.

### localStorage (브라우저 UI 상태)

- **선택 이유**
  - 서버/DB 없이 브라우저 내에서 데이터를 영속적으로 저장하는 표준 Web API입니다.
  - 최대 5~10MB 저장 가능 — 프로젝트 보드 데이터로 충분한 용량입니다.
  - 동기 API(`getItem`/`setItem`)로 단순하게 구현 가능, 비동기 처리 불필요.
  - JSON 직렬화/역직렬화(`JSON.stringify`/`JSON.parse`)로 구조화 데이터 저장 가능.

- **대안과의 비교**

  | 대안 | 탈락 이유 |
  |------|----------|
  | IndexedDB | 비동기 API, 구현 복잡도 높음. 대용량이 아니므로 오버스펙. |
  | sessionStorage | 탭 닫으면 소멸 — 영속성 요구사항 미충족 |
  | Cookie | 4KB 한계, 서버 전송 목적의 API |
  | 서버 DB (PostgreSQL 등) | 서버 필요, 제로 의존성 원칙 위반 |
  | 파일 시스템(브라우저) | File System Access API는 권한 요청 필요, 지원 브라우저 제한 |

- **팀 역량**: 높음 — Web Storage API는 브라우저 기본 API.

---

### JSON 파일 (CLI 상태 저장)

- **선택 이유**
  - `.workflow/state.json`은 CLI가 읽고 쓰는 워크플로우 진행 상태 파일입니다.
  - 사람이 직접 읽고 이해할 수 있는 형식이며, Git으로 버전 관리가 가능합니다.
  - Node.js `fs.readFileSync` / `fs.writeFileSync`로 별도 드라이버 없이 처리합니다.

- **대안과의 비교**

  | 대안 | 탈락 이유 |
  |------|----------|
  | SQLite | 바이너리 파일, Git diff 어렵고 사람이 읽기 불편 |
  | YAML | 파서 라이브러리 필요, JSON보다 생태계 작음 |
  | TOML | Node.js 내장 파서 없음, 패키지 필요 |
  | 환경변수 (.env) | 복잡한 구조화 데이터 표현 불가 |

- **팀 역량**: 높음 — JSON은 보편적 포맷.

## 4. 인프라 / 배포

### GitHub Pages

- **선택 이유**
  - 정적 HTML 파일을 서버 없이 무료로 서빙하는 가장 간단한 방법입니다.
  - `main` 브랜치 루트에 `index.html`이 있으면 즉시 배포됩니다. 별도 설정 불필요.
  - 커스텀 도메인, HTTPS 자동 적용을 무료로 제공합니다.
  - GitHub 저장소와 통합되어 push만으로 자동 배포됩니다.

- **대안과의 비교**

  | 대안 | 장점 | 탈락 이유 |
  |------|------|----------|
  | Netlify | 빌드 파이프라인, 폼 처리, 함수 지원 | 정적 파일만 서빙하는 이 프로젝트에 오버스펙 |
  | Vercel | Next.js 최적화, Edge Functions | 빌드 없는 정적 파일에 오버스펙, Next.js 없음 |
  | Cloudflare Pages | 글로벌 CDN, 빠른 배포 | 정적 파일 서빙에 동일하게 오버스펙 |
  | AWS S3 + CloudFront | 높은 확장성 | 설정 복잡, 비용 발생, 솔로 개발자에게 과도 |

- **팀 역량**: 높음 — GitHub 사용자라면 Pages 설정은 3클릭.

---

### Git + git submodule

- **선택 이유**
  - `.kit/` 폴더에 `claude-commin-kit` 업스트림을 git submodule로 연결합니다.
  - 프로젝트 코드와 툴킷 업스트림을 독립적으로 버전 관리합니다.
  - `git submodule update --remote`로 최신 툴킷 버전을 즉시 적용할 수 있습니다.

- **대안과의 비교**

  | 대안 | 탈락 이유 |
  |------|----------|
  | 파일 복사 | 업스트림 업데이트 시 수동 동기화 필요, 버전 추적 불가 |
  | npm package | 패키지 레지스트리 필요, `package.json` 의존성 발생 |
  | git subtree | submodule보다 명령어 복잡, 독립 업데이트 번거로움 |

- **팀 역량**: 보통 — git submodule 초기화 명령어(`--recurse-submodules`)를 README에 명시로 보완.

## 의도적으로 사용하지 않는 기술

이 프로젝트에서 의도적으로 배제한 기술과 이유:

| 기술 | 배제 이유 |
|------|----------|
| 모든 npm 패키지 | 제로 의존성 원칙 |
| 번들러 (Vite, webpack, Rollup) | 빌드 단계 없는 직접 실행이 목표 |
| TypeScript | 브라우저에서 트랜스파일 없이 실행 불가, Vanilla JS로 충분 |
| 백엔드 서버 (Express, Fastify) | 정적 파일 서빙만 필요 |
| 데이터베이스 (PostgreSQL, MongoDB) | localStorage + JSON으로 충분, 서버리스 원칙 |
| CSS 전처리기 (SCSS, Less) | CSS Custom Properties로 동일 기능 커버 |
| 테스트 프레임워크 | 현재 스프린트 범위 외, 향후 추가 검토 |
