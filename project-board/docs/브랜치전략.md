# Git 브랜치 전략

**프로젝트**: project-board
**작성일**: 2026-03-13

---

## 브랜치 구조

```
main
 └── develop
       ├── feat/기획-스프린트1
       ├── feat/디자인-스프린트1
       ├── feat/프론트-스프린트1
       ├── feat/백엔드-스프린트1
       └── fix/이슈설명
```

---

## 브랜치 정의

| 브랜치 | 용도 | 직접 push |
|--------|------|-----------|
| `main` | 배포용 — GitHub Pages 서빙 기준 | **금지** |
| `develop` | 통합 브랜치 — 모든 feature가 머지되는 곳 | PR 통해서만 |
| `feat/*` | 기능 개발 브랜치 — Phase별로 생성 | 자유롭게 push |
| `fix/*` | 버그 수정 브랜치 | 자유롭게 push |

## feature 브랜치 네이밍

```
feat/{포지션}-스프린트{N}
```

### 예시

```bash
feat/기획-스프린트1
feat/디자인-스프린트1
feat/프론트-스프린트1
feat/백엔드-스프린트1
feat/기획-스프린트2
fix/workflow-상태-초기화-버그
```

## 브랜치 생명주기

```
# 1. develop에서 분기
git checkout develop
git pull origin develop
git checkout -b feat/기획-스프린트1

# 2. 작업 후 커밋
git add docs/
git commit -m "docs: 요구사항, 유저스토리, 차별화 문서 작성"

# 3. 원격에 push
git push origin feat/기획-스프린트1

# 4. GitHub에서 develop으로 PR 생성 → 리뷰 → QC → 머지

# 5. 스프린트 완료 후 develop → main PR 생성 → 머지
```

## 커밋 메시지 컨벤션

### 형식

```
<type>: <subject>

[body]  ← 선택사항

[footer]  ← 선택사항 (이슈 번호 등)
```

### type 종류

| type | 사용 시점 | 예시 |
|------|-----------|------|
| `feat` | 새로운 기능 추가 | `feat: 칸반 보드 드래그 앤 드롭 구현` |
| `fix` | 버그 수정 | `fix: localStorage 상태 복원 오류 수정` |
| `docs` | 문서 작성/수정 | `docs: 요구사항, 유저스토리 문서 작성` |
| `chore` | 빌드, 설정, 의존성 변경 | `chore: GitHub Actions CI 설정 추가` |
| `style` | 코드 포맷, 세미콜론 등 (로직 변경 없음) | `style: 들여쓰기 통일` |
| `refactor` | 기능 변경 없는 코드 개선 | `refactor: 토스트 컴포넌트 추출` |
| `test` | 테스트 추가/수정 | `test: CLI complete 명령어 검증 추가` |

### 작성 규칙

- subject는 **50자 이내**, 마침표 없음
- 한국어 또는 영어 통일 (프로젝트 내 혼용 금지)
- 현재형 동사로 시작 (`추가`, `수정`, `삭제` / `add`, `fix`, `remove`)
- body는 **무엇을**이 아니라 **왜**를 설명

### 올바른 예시

```bash
feat: project-board.html 칸반 5컬럼 레이아웃 추가

Backlog/In Progress/In Review/QC/Done 컬럼 구조를
docs/GitHub-Projects-설정.md 스펙에 맞게 구현

Closes #12
```

```bash
fix: workflow.html 체크박스 상태 새로고침 후 초기화 버그 수정
```

```bash
docs: 브랜치 전략 및 커밋 컨벤션 문서 작성
```

### 잘못된 예시

```bash
# ❌ type 없음
브랜치 전략 추가함

# ❌ 너무 모호함
fix: 수정

# ❌ 과거형
feat: 기능을 추가했습니다.
```

## PR 규칙

- **대상 브랜치**: `feat/*` → `develop`, `develop` → `main`
- **`main` 직접 push 금지** — 반드시 PR을 통해 머지
- PR 제목은 커밋 메시지 컨벤션과 동일한 형식 사용
- 머지 전 조건:
  - [ ] CI 통과 (GitHub Actions)
  - [ ] QC 체크리스트 완료 (★QA 단계 해당)
  - [ ] 리뷰어 1명 이상 승인

## 스프린트별 머지 흐름

```
스프린트 N 시작
  └── feat/기획-스프린트N    → develop (Phase 1 완료 시)
  └── feat/디자인-스프린트N  → develop (Phase 2 완료 시)
  └── feat/프론트-스프린트N  → develop (Phase 3 완료 시)
  └── feat/백엔드-스프린트N  → develop (Phase 4 완료 시)
        ↓ E2E 테스트 통과
  develop → main (스프린트 완료 태그: v1.0.0-sprint{N})
```
